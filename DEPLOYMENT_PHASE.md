# FlowOps HQ Deployment Phase

This is the deployment checklist for taking FlowOps HQ from local MVP to a live hackathon demo.

## 1. Backend on Render

Use `backend/render.yaml` as the Render blueprint.

Required Render environment variables:

- `DATABASE_URL`: PostgreSQL connection URL.
- `REDIS_URL`: Redis connection URL.
- `JWT_SECRET`: generated by Render or a strong custom secret.
- `CORS_ORIGIN`: deployed Vercel frontend URL, for example `https://flowops-dashboard.vercel.app`.
- `OPENAI_API_KEY`: required for OpenAI routing and the Codex prompt optimizer.
- `GEMINI_API_KEY`: required for Gemini routing/fallbacks.
- `ANTHROPIC_API_KEY`: required for Anthropic routing/fallbacks.
- `STRIPE_SECRET_KEY`: optional unless billing checkout is made live.

Render commands:

- Build command: `npm install`
- Start command: `npm run db:push && node src/server.js`

Backend health checks after deploy:

- Open `/health` to confirm the API is alive.
- Open `/metrics` to confirm observability is responding.
- Test auth login/register from the deployed frontend.
- Send one gateway request with `x-flowops-optimize: true` and confirm response headers include `x-flowops-optimized`, `x-flowops-tokens-saved`, and `x-flowops-optimization-percent`.

## 2. Frontend on Vercel

Deploy the `frontend` directory as the Vercel project root.

Required Vercel environment variables:

- `VITE_API_BASE_URL`: Render backend API base URL ending in `/api/v1`, for example `https://flowops-backend.onrender.com/api/v1`.
- `VITE_ENABLE_MOCK_FALLBACK`: set to `false` for production. Use `true` only for a controlled demo fallback.

Vercel build settings:

- Framework preset: Vite
- Build command: `npm run build`
- Output directory: `dist`

Frontend smoke tests after deploy:

- Landing page loads quickly on desktop and mobile.
- Register/login works.
- Workspace routes under `/ws/:slug/...` refresh without 404.
- Dashboard, API keys, analytics, logs, settings, and billing pages render without console errors.
- API calls go to the Render backend, not localhost.

## 3. Demo Readiness

Before submission or judging:

- Run `npm run lint`.
- Run `npm test`.
- Run `npm run build`.
- Keep `VITE_ENABLE_MOCK_FALLBACK=false` for the live product link.
- Keep a backup demo account ready with at least one API route, one key, and sample telemetry.
- Prepare one short optimizer demo using `x-flowops-optimize: true` so judges can see token savings in analytics.
